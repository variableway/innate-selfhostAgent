# Copy-Modify 理念在 AI 场景下的分层实践

> 分析日期：2026-04-16

---

## 什么是 "Copy-Modify"？

Copy-Modify（复制-修改）是编程教育中最经典的学习路径：

```
Step 1: Copy（复制）
    拿到一个能跑起来的完整案例
    ├── 确认它确实能工作
    └── 理解它的大致结构
    
Step 2: Modify（修改）
    在不破坏整体框架的前提下做小幅改动
    ├── 改一个颜色 / 一个数字 / 一个名字
    ├── 观察改动带来的结果变化
    └── 建立"输入 → 输出"的因果直觉
    
Step 3: Remix（重组）
    把多个案例的模块拼接起来
    ├── A 的触发器 + B 的处理逻辑 + C 的输出格式
    └── 创造属于自己的版本
    
Step 4: Create（原创）
    从零开始构建自己的作品
    └── 此时脑海中已经积累了足够的"可调用模块"
```

**在 AI 时代，这个路径变得更加重要**——因为 AI 降低了"复制"的门槛，但也让人更容易跳过"修改"和"理解"的阶段，直接要求 AI 生成完整方案，结果拿到手却不知道怎么改。

**Copy-Modify 的核心价值：让学习者有"掌控感"——不是被 AI 的输出生硬地喂养，而是通过小幅修改看到"我也能让它变得更好"。**

---

## 分层策略：小白 vs 进阶用户

### 对小白用户：降低修改成本，放大反馈速度

**核心痛点**：
- 拿到一个 Prompt/工作流，不知道从哪里改起
- 怕改坏了，不敢动手
- 改了之后没看出区别，没有成就感

**Copy-Modify 设计原则**：

| 原则 | 说明 |
|------|------|
| **原子化复制单元** | 每次只给一个小而完整的可复制对象（一个 Prompt、一个按钮、一个命令） |
| **高亮可修改区** | 用 `【】` 或 `{{}}` 明确标出"你只需要改这里" |
| **即时反馈** | 修改后 3 秒内就能看到输出变化 |
| **容错设计** | 提供"恢复默认"按钮，让小白敢试错 |
| **渐进暴露** | 从改 1 个词 → 改 1 句话 → 改 1 个段落 → 改结构 |

---

### 对进阶用户：模块化拆解，建立个人武器库

**核心痛点**：
- 网上的 Prompt 太通用，不满足专业需求
- 有自己的工作流，但不知道怎么和 AI 能力拼接
- 积累了大量碎片，缺乏系统化的复用框架

**Copy-Modify 设计原则**：

| 原则 | 说明 |
|------|------|
| **模块化结构** | 明确标出 Prompt 的"角色-任务-上下文-约束-输出格式"等模块 |
| **参数化设计** | 把常用变量提取成可配置项（temperature、style、depth） |
| **版本管理** | 支持保存多个修改版本，对比效果差异 |
| **反编译学习** | 把优秀的 AI 输出拆解回 Prompt 结构，学习别人的设计思路 |
| **组合创新** | 提供模块库，让用户像拼乐高一样组装新 Prompt |

## 场景一：Prompt 的 Copy-Modify

### 小白版：填空式 Prompt

**原始 Prompt（完整可复制）**：
```
请帮我写一段产品经理的自我介绍，要求：
- 突出 3 年工作经验
- 提到擅长的领域是用户增长
- 语气专业但不生硬
- 字数在 150 字左右
```

**Copy-Modify 版（高亮可修改区）**：
```
请帮我写一段【产品经理】的自我介绍，要求：
- 突出【3 年】工作经验
- 提到擅长的领域是【用户增长】
- 语气【专业但不生硬】
- 字数在【150】字左右
```

**小白操作路径**：
1. 复制完整 Prompt
2. 把 `【产品经理】` 改成 `【UI 设计师】`
3. 把 `【3 年】` 改成 `【5 年】`
4. 把 `【用户增长】` 改成 `【视觉系统搭建】`
5. 发送，看到新输出

**成就感来源**："我只改了 3 个词，就得到了完全适合我的自我介绍。"

---

### 进阶版：模块化 Prompt 模板

**原始结构（可被复制和深度修改）**：
```markdown
# 角色设定（Role）
你是一位【领域】的资深专家，拥有【N】年实战经验，
擅长【技能 A】和【技能 B】，风格偏向【风格描述】。

# 任务描述（Task）
请为【目标对象】撰写一份【文档类型】，
核心目的是【目的】，需要突出的亮点是【亮点】。

# 约束条件（Constraints）
- 字数：【X】字左右
- 语气：【语气描述】
- 必须包含的要素：【要素 1, 要素 2, 要素 3】
- 避免使用的表达：【禁忌词】

# 输出格式（Output Format）
【指定格式：Markdown / JSON / 表格 / 分段结构】

# 参考示例（Few-shot）
【如果需要，粘贴 1-2 个参考案例】

# 上下文信息（Context）
【补充背景：公司阶段、目标受众、竞争环境等】
```

**进阶用户操作路径**：
1. 复制这个结构到自己的 Prompt Library
2. 根据任务类型，替换各个模块的内容
3. 保存为"产品文案模板_v1"
4. 下次写技术博客时，只改 Role 和 Task，约束条件复用
5. 对比不同 Few-shot 示例对输出质量的影响

## 场景二：可执行命令/终端脚本的 Copy-Modify

### 小白版：一键运行 + 改参数

**在 Innate Playground 中的设计**：

```bash
# 原始命令（可复制，确认能跑）
$ python scripts/photo_diagnose.py \
    --input ~/Photos/example.jpg \
    --style detailed

# Copy-Modify 版（高亮可修改区）
$ python scripts/photo_diagnose.py \
    --input 【你的照片路径】 \
    --style 【detailed 或 simple】
```

**小白操作路径**：
1. 点击 Playground 中的 "RunButton"，先看默认示例跑通
2. 复制命令到终端
3. 把 `~/Photos/example.jpg` 改成自己电脑上的真实路径
4. 把 `detailed` 改成 `simple` 试试效果差异
5. 运行，观察输出变化

**成就感来源**："我只改了一个路径，就让 AI 分析了我自己的照片。"

---

### 进阶版：脚本模块化改造

**原始脚本结构（Python）**：
```python
# 1. 配置区（可复制后修改）
INPUT_DIR = "~/Photos"
OUTPUT_DIR = "~/Reports"
MODEL = "gpt-4o"
STYLE = "detailed"

# 2. 核心处理逻辑（通常不需要改）
def batch_process(input_dir, output_dir, model, style):
    for file in os.listdir(input_dir):
        if file.endswith((".jpg", ".png")):
            result = analyze(file, model=model, style=style)
            save(result, output_dir)

# 3. 执行入口
batch_process(INPUT_DIR, OUTPUT_DIR, MODEL, STYLE)
```

**进阶用户操作路径**：
1. 复制整个脚本
2. 在"配置区"改参数：批量处理 `~/Design` 文件夹里的 `.psd` 文件
3. 在"核心逻辑区"加一个小模块：只处理文件名包含 `"final"` 的文件
4. 保存为 `design_review.py`，成为个人工作流的一部分
5. 进一步把"配置区"提取到 `config.yaml`，让脚本更通用

## 场景三：AI 工作流的 Copy-Modify

### 小白版：模板化工作流

**原始工作流（摄影分析）**：
```
Step 1: 上传照片
Step 2: AI 提取 EXIF
Step 3: AI 分析构图
Step 4: AI 生成后期建议
Step 5: 输出报告
```

**Copy-Modify 版**：
```
这个工作流可以处理：【照片 / 设计稿 / 穿搭】
分析维度包括：【构图 / 色彩 / 风格 / 技术参数】
输出格式：【详细报告 / 一句话总结 / 对比图】
```

**小白操作路径**：
1. 先完整跑一遍"照片分析"工作流（Copy）
2. 在 Playground 的技能市场中找到"穿搭分析"版本
3. 对比两个工作流的差异：Step 2 从"提取 EXIF"变成了"识别单品和品牌"
4. 意识到"只要替换中间的分析模块，就能处理新类型"
5. 下次想分析"室内设计图"时，尝试把 Step 2 改成"识别空间和家具"

---

### 进阶版：工作流编排平台

**进阶用户的设计**：
```yaml
workflow_name: "我的内容生产流水线"
version: 1.2

steps:
  - name: "抓取"
    tool: "web_scraper"
    config:
      urls: "{{input_urls}}"
      
  - name: "摘要"
    tool: "llm_summarize"
    config:
      model: "gpt-4o"
      style: "{{summary_style}}"  # 可修改：bullet / narrative / tweet
      
  - name: "分类"
    tool: "llm_classify"
    config:
      categories: "{{custom_categories}}"  # 可修改
      
  - name: "输出"
    tool: "notion_exporter"
    config:
      database_id: "{{notion_db_id}}"
```

**进阶用户操作路径**：
1. 复制一个别人的工作流 YAML
2. 把 `web_scraper` 换成自己的 RSS 订阅源
3. 把 `notion_exporter` 换成 `feishu_doc_exporter`
4. 在"摘要"和"分类"之间插入一个新步骤："情感倾向分析"
5. 保存为个人版本，定时运行

## 场景四：Skill 的 Copy-Modify

### 小白版：改参数即可用的技能包

**Innate Playground 的技能设计**：

每个 Skill 都提供：
1. **标准版**：直接运行
2. **Copy-Modify 版**：标注出 3-5 个可修改的参数

**示例：「明日三餐规划」Skill**

```yaml
skill: meal_planner_v1

# 可修改参数（Copy-Modify 区）
params:
  diet_goal: "减脂"        # 可改：增肌 / 维持 / 控糖
  meal_count: 3            # 可改：2 / 3 / 4 / 5
  cuisine_pref: "中餐"     # 可改：西餐 / 日料 / 混合
  time_limit: "30分钟"     # 可改：15分钟 / 30分钟 / 不限
  allergy: "无"            # 可改：花生 / 海鲜 / 乳糖不耐
```

**小白操作路径**：
1. 先运行默认版，看看输出长什么样
2. 打开"参数面板"，把 `diet_goal` 从"减脂"改成"增肌"
3. 把 `time_limit` 改成"15分钟"
4. 再次运行，对比两份菜单的差异
5. 保存为自己的"增肌快手菜"配置

---

### 进阶版：Skill 的源码级改造

**进阶用户的 Copy-Modify 路径**：

```bash
# 1. 复制官方 Skill 到个人目录
$ cp -r skills/official/meal_planner skills/personal/meal_planner_pro

# 2. 修改核心逻辑
# 比如把基于规则的菜谱推荐，改成调用下厨房 API
$ vim skills/personal/meal_planner_core.py

# 3. 添加新模块
# 比如加入"根据当日运动强度调整热量"的功能
$ vim skills/personal/workout_calorie_adjuster.py

# 4. 修改元数据和导出配置
$ vim skills/personal/skill_manifest.yaml

# 5. 测试并发布到个人 Skill 库
$ python scripts/test_skill.py --skill meal_planner_pro
```

**成就感来源**："我不只是用别人的 Skill，我在它的基础上造出了更适合我的版本。"

## 场景五：从 AI 输出反推 Prompt（Reverse Engineering）

### 小白版："这个结果不错，它是怎么来的？"

**产品功能设计**：在 Playground 中，每个 AI 输出卡片底部有一个小按钮：
> `[👁️ 查看生成这个结果的 Prompt]`

点击后展开：
```markdown
这个结果是基于以下 Prompt 生成的：

---
角色：你是一位专业摄影师...
任务：请分析这张照片的构图...
约束：用中文回答，避免黑话...
---

你可以复制这个 Prompt，然后修改【照片路径】来分析自己的照片。
```

**小白学习路径**：
1. 先用 RunButton 跑出一个好结果
2. 点击"查看 Prompt"，复制完整 Prompt
3. 在记事本里改几个参数
4. 直接在 ChatGPT/Claude 里粘贴测试
5. 对比修改前后的输出差异

---

### 进阶版：Prompt 反编译与结构学习

**进阶用户的操作**：

看到一段优秀的 AI 输出后，尝试还原它的 Prompt 结构：

```markdown
【观察到的输出特征】
- 语气：理性但有温度
- 结构：问题定义 → 多角度分析 →  actionable 建议 → 一句金句结尾
- 深度：每个角度都有 2-3 层递进
- 用词：没有陈词滥调，有具体场景描写

【推测的 Prompt 结构】
1. 角色：资深心理咨询师 + 资深产品经理的 hybrid
2. 任务：对用户的情感/职业困境进行结构化分析
3. 约束：
   - 每个角度必须包含：洞察 + 反直觉观察 + 具体行动
   - 结尾必须有一句 20 字以内的金句
   - 避免使用"首先/其次/最后"的过渡词
4. 上下文：提供了用户的背景信息（年龄/职业/近期事件）
5. Few-shot：可能给了 1-2 个类似风格的参考案例
```

**训练目标**：通过大量反编译，建立对"好 Prompt 结构"的肌肉记忆。

## 不同用户的 Copy-Modify 成长路径

### 小白用户（0-3 个月）

```
Week 1-2: 纯复制（Copy Only）
  → 找到好用的 Prompt，直接复制，不改任何内容
  → 目标是：确认"这东西确实能工作"

Week 3-4: 填空式修改（Fill-in-the-blanks）
  → 改 `【】` 里的内容（名字、数字、领域）
  → 观察输出变化，建立"改 A 导致 B"的直觉

Month 2-3: 组合式复制（Copy & Mix）
  → 把 Prompt A 的角色设定 + Prompt B 的输出格式拼在一起
  → 尝试把 Playground 的 RunButton 命令改路径后运行
```

### 进阶用户（3-12 个月）

```
Month 3-4: 模块化修改（Modular Modify）
  → 理解 Prompt 的 Role-Task-Constraint-Format 结构
  → 有意识地替换/增删模块

Month 5-6: 参数化设计（Parametric Design）
  → 把常用的 Prompt 变成模板（用变量代替固定值）
  → 建立个人的 Prompt 库和脚本库

Month 7-9: 反编译学习（Reverse Engineering）
  → 看到好输出，能推测背后的 Prompt 设计思路
  → 把别人的优秀工作流拆解并本地化改造

Month 10-12: 原创构建（Create）
  → 从零设计完整的 Skill 或工作流
  → 把多个 AI 能力拼接成解决复杂问题的个人系统
```

## 关键设计原则

### 对小白

1. **每次只改一个地方**
   如果让用户同时改 5 个参数，出错后他不知道哪个参数导致了问题。

2. **提供"前后对比"视图**
   左边是默认输出，右边是修改后的输出，差异一目了然。

3. **设计"不会坏"的修改**
   比如先让用户改"语气词"（从"专业"到"幽默"），而不是改"整个逻辑结构"。

4. **用游戏化降低试错恐惧**
   "试试把温度参数调到 2.0，看看 AI 会不会发疯"
   → 把"试错"包装成"实验"和"探索"。

### 对进阶用户

1. **明确标注"稳定区"和"可改区"**
   ```
   # ===== 稳定区：通常不需要修改 =====
   def core_logic():
       ...
   
   # ===== 可改区：根据你的需求调整 =====
   CONFIG = {
       "model": "gpt-4o",
       "style": "detailed"
   }
   ```

2. **提供版本 diff 工具**
   让用户能看到"官方版 v1.2"和"我的修改版"之间的差异。

3. **鼓励" forks "文化**
   "这个 Skill 已经有 47 个用户 fork 了自己的版本，点击查看他们的创新点。"

4. **建立模块化市场**
   不是卖完整的 Skill，而是卖可替换的模块：
   - "财务分析的图表生成模块"
   - "会议纪要的情绪识别模块"
   - "摄影分析的胶片模拟点评模块"

## 与 Innate Playground 的产品融合

### 功能 1："一键 Fork"

每个官方 Skill 都提供：
- `[运行官方版]`
- `[Fork 到我的空间]`

Fork 后，用户进入"修改模式"：
```
┌─────────────────────────────────────────┐
│  正在编辑：meal_planner（个人版）        │
├─────────────────────────────────────────┤
│  左侧：官方原版代码/配置                 │
│  右侧：你的修改版                        │
│  绿色：你新增的内容                      │
│  红色：你删除的内容                      │
│  黄色：官方已更新的内容                  │
├─────────────────────────────────────────┤
│  [保存] [测试运行] [发布到个人库]        │
│  [对比效果] [查看修改历史]               │
└─────────────────────────────────────────┘
```

### 功能 2："填空式 Prompt 工坊"

一个专门的练习页面：
```
练习 1：改一个参数
原始：请用【专业】的语气写一段自我介绍
修改：请用【____】的语气写一段自我介绍
      ↓ 下拉选择：幽默 / 诗意 / 激进 / 温柔
      
[运行并对比] → 弹出左右分屏的输出对比
```

### 功能 3："模块拼装台"

```
┌─────────────────────────────────────────┐
│  🧩 模块拼装台                          │
├─────────────────────────────────────────┤
│  已选模块：                              │
│  [角色设定] → 摄影师                     │
│  [输入处理] → 单张照片                   │
│  [分析引擎] → 构图分析模块               │
│  [输出格式] → Markdown 报告              │
│  [附加模块] → 后期参数建议模块 (+)       │
│                                          │
│  [生成完整 Prompt / 命令]               │
│  [保存为自定义 Skill]                    │
└─────────────────────────────────────────┘
```

## 一句话总结

> **Copy-Modify 是 AI 时代最重要的学习元技能。对小白，它是"改一个词就能看到变化"的即时成就感；对进阶用户，它是"理解结构、模块化改造、建立个人武器库"的系统化能力。Innate Playground 的核心使命，就是在这两个层级上都提供完美的 Copy-Modify 体验——让每个人都能从"用别人的 AI"进化到"造自己的 AI"。**
